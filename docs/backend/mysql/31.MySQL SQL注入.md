## 1.引言

SQL 注入是一种常见的数据库安全漏洞，当应用程序将用户输入直接拼接到 SQL 语句中执行时，攻击者可以通过构造恶意输入来执行非预期的 SQL 命令，从而获取、修改或删除数据库中的数据。

本文介绍 SQL 注入的原理、危害以及如何防范 SQL 注入攻击。通过实际示例，你将了解 SQL 注入是如何发生的，以及如何通过参数化查询、输入验证等方法保护数据库安全。

## 2.SQL 注入的原理

### 2.1.什么是 SQL 注入？

SQL 注入是指攻击者通过在用户输入中插入恶意的 SQL 代码，使得应用程序执行非预期的 SQL 查询。这通常发生在应用程序直接将用户输入拼接到 SQL 语句中，而没有进行适当的验证和转义。

**SQL 注入的危害**：

- 获取敏感数据（用户信息、密码等）
- 修改或删除数据库中的数据
- 绕过身份验证，获得未授权访问
- 执行系统命令，控制服务器

### 2.2.SQL 注入是如何发生的？

假设有一个商品搜索功能，用户输入商品名称来查询商品信息：


```sql
SELECT * FROM products WHERE product_name = '用户输入的商品名';
```

如果应用程序直接将用户输入拼接到 SQL 语句中，没有进行任何处理：


```sql
SELECT * FROM products WHERE product_name = '{$user_input}';
```

当用户输入正常的商品名时，查询正常执行。但如果攻击者输入恶意内容，就可能改变 SQL 语句的逻辑。

### 2.3.SQL 注入示例

**正常查询**：

用户输入：`笔记本电脑`

生成的 SQL：


```sql
SELECT * FROM products WHERE product_name = '笔记本电脑';
```

**恶意注入**：

攻击者输入：`' OR '1'='1' --`

生成的 SQL：


```sql
SELECT * FROM products WHERE product_name = '' OR '1'='1' --';
```

**分析**：

- `'` 闭合了原来的单引号
- `OR '1'='1'` 是一个永远为真的条件
- `--` 是 SQL 注释符号，会注释掉后面的内容

这个查询会返回所有商品，因为 `'1'='1'` 永远为真。

## 3.常见的 SQL 注入场景

### 3.1.用户身份验证绕过

假设有一个用户登录验证的 SQL 查询：


```sql
SELECT * FROM accounts WHERE user_id = '输入的用户名' AND password = '输入的密码';
```

如果攻击者输入以下内容：

- 用户名：`admin' --`
- 密码：任意值

生成的 SQL：


```sql
SELECT * FROM accounts WHERE user_id = 'admin' --' AND password = '任意值';
```

由于 `--` 注释掉了后面的密码验证，只要用户名 `admin` 存在，就能成功登录，完全绕过了密码验证。

### 3.2.数据泄露

攻击者可以通过 UNION 查询获取其他表的数据：


```sql
-- 原始查询
SELECT product_name, price FROM products WHERE category = '电子产品';

-- 注入后
SELECT product_name, price FROM products WHERE category = '电子产品' UNION SELECT username, password FROM users --';
```

这样就能获取用户表中的敏感信息。

### 3.3.数据删除

攻击者可以注入 DELETE 语句来删除数据：


```sql
-- 原始查询
SELECT * FROM products WHERE product_name = '用户输入';

-- 注入后
SELECT * FROM products WHERE product_name = 'test'; DROP TABLE products; --';
```

虽然 MySQL 的 `mysqli_query()` 默认不允许执行多条语句，但在某些配置下或使用其他数据库时，这可能导致整个表被删除。

## 4.防范 SQL 注入的方法

### 4.1.使用参数化查询（推荐）

参数化查询（Prepared Statements）是最有效的防范 SQL 注入的方法。它将 SQL 语句的结构与数据分离，数据库会先编译 SQL 语句，然后再将参数值绑定到占位符上。

**参数化查询的优势**：

- 数据库会正确处理特殊字符，自动转义
- SQL 语句结构固定，无法被注入代码改变
- 性能更好，可以预编译和重用

**示例：使用参数化查询**


```python
# Python 示例
import mysql.connector

conn = mysql.connector.connect(
    host='localhost',
    user='root',
    password='password',
    database='mydb'
)

cursor = conn.cursor()

# 使用参数化查询
query = "SELECT * FROM products WHERE product_name = %s"
product_name = user_input  # 用户输入
cursor.execute(query, (product_name,))

results = cursor.fetchall()
```

### 4.2.输入验证和过滤

对用户输入进行严格的验证，只允许符合预期格式的数据通过。

**验证规则示例**：


```python
import re

def validate_product_name(name):
    # 只允许字母、数字、中文字符和空格，长度 1-50
    pattern = r'^[\w\s\u4e00-\u9fa5]{1,50}$'
    if re.match(pattern, name):
        return True
    return False

# 使用验证
if validate_product_name(user_input):
    # 执行查询
    pass
else:
    print("输入格式不正确")
```

**验证要点**：

- 检查数据类型（数字、字符串等）
- 检查长度范围
- 检查字符集（只允许特定字符）
- 检查格式（邮箱、电话等）

### 4.3.转义特殊字符

如果必须使用字符串拼接，需要对特殊字符进行转义。

**注意**：转义方法不如参数化查询安全，应优先使用参数化查询。

### 4.4.最小权限原则

为数据库用户分配最小必要的权限，限制其只能执行必要的操作。

**权限设置示例**：


```sql
-- 创建只读用户
CREATE USER 'readonly_user'@'localhost' IDENTIFIED BY 'password';
GRANT SELECT ON mydb.products TO 'readonly_user'@'localhost';

-- 创建应用用户，只能执行特定操作
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'password';
GRANT SELECT, INSERT, UPDATE ON mydb.products TO 'app_user'@'localhost';
-- 不授予 DELETE 权限，防止数据被删除
```

### 4.5.禁用错误信息显示

在生产环境中，不要向用户显示详细的数据库错误信息，这些信息可能泄露数据库结构。

**错误处理示例**：


```python
try:
    cursor.execute(query, params)
except mysql.connector.Error as err:
    # 记录详细错误到日志
    logger.error(f"Database error: {err}")
    # 向用户显示通用错误信息
    return {"error": "操作失败，请稍后重试"}
```

## 5.LIKE 查询中的 SQL 注入

### 5.1.LIKE 查询的特殊字符

在 LIKE 查询中，`%` 和 `_` 是通配符，具有特殊含义：

- `%`：匹配任意多个字符
- `_`：匹配单个字符

如果用户输入包含这些字符，可能导致意外的查询结果。

**问题示例**：


```sql
-- 用户想搜索 "test_123"
SELECT * FROM products WHERE product_name LIKE 'test_123%';

-- 实际匹配了 "testA123"、"testB123" 等，因为 _ 匹配了任意单个字符
```

### 5.2.转义 LIKE 查询中的特殊字符

需要对 `%` 和 `_` 进行转义，使其作为普通字符处理。

**Python 示例**：


```python
import re

def escape_like_pattern(pattern):
    # 转义 % 和 _
    pattern = pattern.replace('\\', '\\\\')  # 先转义反斜杠
    pattern = pattern.replace('%', '\\%')
    pattern = pattern.replace('_', '\\_')
    return pattern

search_term = escape_like_pattern(user_input)
query = "SELECT * FROM products WHERE product_name LIKE %s"
cursor.execute(query, (f"{search_term}%",))
```

## 6.防范 SQL 注入的最佳实践

### 6.1.永远不要信任用户输入

将所有用户输入视为不可信的，必须进行验证和过滤：


```python
def sanitize_input(user_input):
    # 1. 验证数据类型
    if not isinstance(user_input, str):
        raise ValueError("输入必须是字符串")
    
    # 2. 验证长度
    if len(user_input) > 100:
        raise ValueError("输入长度超出限制")
    
    # 3. 验证字符集
    if not user_input.isalnum() and '_' not in user_input:
        raise ValueError("输入包含非法字符")
    
    return user_input
```

### 6.2.使用参数化查询

优先使用参数化查询，避免字符串拼接：


```python
# 正确：使用参数化查询
cursor.execute("SELECT * FROM products WHERE id = %s", (product_id,))

# 错误：字符串拼接（存在注入风险）
cursor.execute(f"SELECT * FROM products WHERE id = {product_id}")
```

### 6.3.使用 ORM 框架

使用对象关系映射（ORM）框架可以自动处理 SQL 查询，降低 SQL 注入的风险：


```python
# 使用 SQLAlchemy ORM（Python）
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

# ORM 会自动使用参数化查询
user = session.query(User).filter(User.username == user_input).first()
```
