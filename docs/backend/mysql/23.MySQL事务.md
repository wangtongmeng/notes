## 1.引言

在数据库操作中，我们经常需要执行一系列相关的操作，这些操作要么全部成功，要么全部失败。比如在电商系统中，用户下单时需要同时更新库存、创建订单、扣除账户余额等多个操作，如果其中任何一个操作失败，整个流程都应该回滚，保证数据的一致性。MySQL 事务就是用来实现这个功能的。

本文将从事务的基础概念开始，逐步介绍 MySQL 事务的使用方法，包括事务的 ACID 特性、事务控制语句、事务的开启和提交，以及如何在实际应用中使用事务来保证数据的完整性。

## 2.事务基础概念

### 2.1.什么是事务？

事务（Transaction）是数据库操作的一个逻辑单元，它由一系列 SQL 语句组成，这些语句作为一个整体要么全部执行成功，要么全部不执行。

**类比理解**：

- 事务就像银行转账操作
- 转账需要两个步骤：从 A 账户扣款，向 B 账户加款
- 这两个步骤必须同时成功或同时失败，不能只完成其中一个
- 如果只扣了 A 账户的钱，但 B 账户没有收到，就会造成数据不一致

### 2.2.事务的应用场景

事务主要用于处理需要保证数据一致性的操作场景：

- **转账操作**：从一个账户扣款，同时向另一个账户加款
- **订单处理**：创建订单、更新库存、扣除余额等多个操作
- **数据删除**：删除主记录时，同时删除相关的子记录
- **批量更新**：需要同时更新多个相关表的数据

### 2.3.事务的存储引擎要求

在 MySQL 中，只有使用 InnoDB 存储引擎的表才支持事务。MyISAM 存储引擎不支持事务。

**查看表的存储引擎**：


```sql
-- 查看表的存储引擎类型
SHOW TABLE STATUS WHERE Name = 'table_name';
```

**创建支持事务的表**：


```sql
-- 创建使用 InnoDB 引擎的表
CREATE TABLE example_table (
    id INT PRIMARY KEY,
    name VARCHAR(50)
) ENGINE=InnoDB;
```

## 3.事务的 ACID 特性

事务必须满足四个基本特性，简称 ACID：

### 3.1.原子性（Atomicity）

原子性是指事务中的所有操作要么全部完成，要么全部不完成，不会出现部分完成的情况。

**理解**：

- 就像原子不可分割一样，事务也不可分割
- 如果事务执行过程中发生错误，所有已执行的操作都会被撤销
- 数据库会回滚到事务开始前的状态

**示例场景**：

- 转账操作中，如果扣款成功但加款失败，整个事务会回滚，扣款操作也会被撤销

### 3.2.一致性（Consistency）

一致性是指事务执行前后，数据库的状态必须保持一致，不会破坏数据库的完整性约束。

**理解**：

- 事务执行后，数据库从一个一致状态转换到另一个一致状态
- 所有数据必须符合预设的规则和约束
- 不会出现数据矛盾或不完整的情况

**示例场景**：

- 转账后，两个账户的总金额应该保持不变
- 订单创建后，库存数量应该正确减少

### 3.3.隔离性（Isolation）

隔离性是指多个事务并发执行时，每个事务都感觉不到其他事务的存在，就像只有自己在操作数据库一样。

**理解**：

- 并发执行的事务之间相互隔离
- 一个事务的中间状态不会被其他事务看到
- 防止并发操作导致的数据不一致问题

**隔离级别**：

- **读未提交（READ UNCOMMITTED）**：最低级别，一个事务可以读取另一个未提交事务的数据
- **读已提交（READ COMMITTED）**：只能读取已提交事务的数据
- **可重复读（REPEATABLE READ）**：MySQL 默认级别，保证同一事务中多次读取结果一致
- **串行化（SERIALIZABLE）**：最高级别，事务串行执行，完全隔离

### 3.4.持久性（Durability）

持久性是指事务一旦提交，对数据库的修改就是永久性的，即使系统发生故障也不会丢失。

**理解**：

- 提交后的数据会持久保存到磁盘
- 系统崩溃或断电不会影响已提交的数据
- 数据库恢复后，已提交的事务结果仍然存在

## 4.MySQL 事务控制语句

### 4.1.开启事务

在 MySQL 中，可以使用以下方式开启一个事务：

**方式一：使用 BEGIN 语句**


```sql
-- 开始一个新事务
BEGIN;
```

**方式二：使用 START TRANSACTION 语句**


```sql
-- 开始一个新事务
START TRANSACTION;
```

**方式三：关闭自动提交**


```sql
-- 关闭当前会话的自动提交模式
SET AUTOCOMMIT = 0;
```

**说明**：

- `BEGIN` 和 `START TRANSACTION` 功能相同，都是显式开启一个事务
- `SET AUTOCOMMIT = 0` 会关闭自动提交，之后的所有 SQL 语句都需要手动提交
- MySQL 默认情况下，每条 SQL 语句都会自动提交（AUTOCOMMIT = 1）

### 4.2.提交事务

提交事务会将事务中的所有操作永久保存到数据库：


```sql
-- 提交当前事务
COMMIT;
```

**或者使用**：


```sql
-- 提交当前事务（与 COMMIT 等价）
COMMIT WORK;
```

**说明**：

- `COMMIT` 会提交事务，使所有修改成为永久性的
- 提交后，事务中的所有操作都会生效
- 提交后无法再回滚

### 4.3.回滚事务

回滚事务会撤销事务中的所有操作，恢复到事务开始前的状态：


```sql
-- 回滚当前事务
ROLLBACK;
```

**或者使用**：


```sql
-- 回滚当前事务（与 ROLLBACK 等价）
ROLLBACK WORK;
```

**说明**：

- `ROLLBACK` 会撤销事务中的所有未提交的修改
- 回滚后，数据库恢复到事务开始前的状态
- 只能回滚未提交的事务

### 4.4.保存点（SAVEPOINT）

保存点允许在事务中创建一个标记点，可以回滚到该点而不是回滚整个事务：

**创建保存点**：


```sql
-- 在事务中创建一个保存点
SAVEPOINT savepoint_name;
```

**回滚到保存点**：


```sql
-- 回滚到指定的保存点
ROLLBACK TO SAVEPOINT savepoint_name;
```

**删除保存点**：


```sql
-- 删除指定的保存点
RELEASE SAVEPOINT savepoint_name;
```

**说明**：

- 一个事务中可以创建多个保存点
- 回滚到保存点后，该保存点之后的操作会被撤销
- 保存点在事务提交或回滚后会自动删除

### 4.5.设置事务隔离级别

可以使用 `SET TRANSACTION` 语句设置事务的隔离级别：


```sql
-- 设置事务隔离级别为读未提交
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 设置事务隔离级别为读已提交
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置事务隔离级别为可重复读
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 设置事务隔离级别为串行化
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

**说明**：

- 隔离级别影响事务的并发行为
- 隔离级别越高，数据一致性越好，但性能可能下降
- MySQL 默认使用 REPEATABLE READ 隔离级别

## 5.事务使用示例

### 5.1.创建测试表

首先创建一个用于测试的表：


```sql
-- 创建商品库存表
CREATE TABLE product_inventory (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(50),
    stock_quantity INT,
    price DECIMAL(10, 2)
) ENGINE=InnoDB;

-- 插入测试数据
INSERT INTO product_inventory VALUES
(1, '商品A', 100, 50.00),
(2, '商品B', 200, 80.00),
(3, '商品C', 150, 120.00);
```

### 5.2.基本事务示例：提交事务

**场景**：更新商品库存和价格


```sql
-- 开始事务
START TRANSACTION;

-- 更新商品A的库存
UPDATE product_inventory 
SET stock_quantity = stock_quantity - 10 
WHERE product_id = 1;

-- 更新商品A的价格
UPDATE product_inventory 
SET price = 55.00 
WHERE product_id = 1;

-- 提交事务，使所有修改生效
COMMIT;
```

**代码说明**：

- `START TRANSACTION`：开始一个新事务
- `UPDATE product_inventory SET stock_quantity = stock_quantity - 10 WHERE product_id = 1`：将商品A的库存减少10
- `UPDATE product_inventory SET price = 55.00 WHERE product_id = 1`：将商品A的价格更新为55.00
- `COMMIT`：提交事务，使所有修改永久生效

**执行结果**：

- 如果两个 UPDATE 语句都执行成功，COMMIT 后所有修改都会保存
- 如果任何一个 UPDATE 失败，可以执行 ROLLBACK 撤销所有操作

### 5.3.基本事务示例：回滚事务

**场景**：尝试更新商品信息，如果出现错误则回滚


```sql
-- 开始事务
BEGIN;

-- 插入新商品
INSERT INTO product_inventory VALUES (4, '商品D', 300, 90.00);

-- 更新商品B的库存（假设这里出现了问题）
UPDATE product_inventory 
SET stock_quantity = stock_quantity - 500 
WHERE product_id = 2;

-- 检查发现库存不能为负数，回滚事务
ROLLBACK;
```

**代码说明**：

- `BEGIN`：开始一个新事务
- `INSERT INTO product_inventory VALUES (4, '商品D', 300, 90.00)`：插入一条新商品记录
- `UPDATE product_inventory SET stock_quantity = stock_quantity - 500 WHERE product_id = 2`：尝试将商品B的库存减少500（但商品B只有200库存）
- `ROLLBACK`：回滚事务，撤销所有未提交的操作

**执行结果**：

- 回滚后，新插入的商品D记录会被撤销
- 商品B的库存更新也会被撤销
- 数据库恢复到事务开始前的状态

### 5.4.使用保存点示例

**场景**：在事务中使用保存点，部分回滚


```sql
-- 开始事务
START TRANSACTION;

-- 更新商品A的库存
UPDATE product_inventory 
SET stock_quantity = stock_quantity - 5 
WHERE product_id = 1;

-- 创建保存点
SAVEPOINT sp1;

-- 更新商品B的库存
UPDATE product_inventory 
SET stock_quantity = stock_quantity - 20 
WHERE product_id = 2;

-- 创建另一个保存点
SAVEPOINT sp2;

-- 更新商品C的库存
UPDATE product_inventory 
SET stock_quantity = stock_quantity - 15 
WHERE product_id = 3;

-- 发现商品C的更新有问题，回滚到保存点sp2
ROLLBACK TO SAVEPOINT sp2;

-- 提交事务
COMMIT;
```

**代码说明**：

- `START TRANSACTION`：开始一个新事务
- `UPDATE product_inventory SET stock_quantity = stock_quantity - 5 WHERE product_id = 1`：更新商品A的库存
- `SAVEPOINT sp1`：创建第一个保存点
- `UPDATE product_inventory SET stock_quantity = stock_quantity - 20 WHERE product_id = 2`：更新商品B的库存
- `SAVEPOINT sp2`：创建第二个保存点
- `UPDATE product_inventory SET stock_quantity = stock_quantity - 15 WHERE product_id = 3`：更新商品C的库存
- `ROLLBACK TO SAVEPOINT sp2`：回滚到保存点sp2，撤销商品C的更新
- `COMMIT`：提交事务，商品A和商品B的更新会生效

**执行结果**：

- 商品A的库存更新会生效（减少5）
- 商品B的库存更新会生效（减少20）
- 商品C的库存更新会被撤销（因为回滚到了sp2保存点）

### 5.5.转账场景示例

**场景**：模拟银行转账，从账户A向账户B转账

首先创建账户表：


```sql
-- 创建账户表
CREATE TABLE bank_account (
    account_id INT PRIMARY KEY,
    account_name VARCHAR(50),
    balance DECIMAL(10, 2)
) ENGINE=InnoDB;

-- 插入测试数据
INSERT INTO bank_account VALUES
(1, '账户A', 1000.00),
(2, '账户B', 500.00);
```

**转账操作**：


```sql
-- 开始事务
BEGIN;

-- 从账户A扣除100元
UPDATE bank_account 
SET balance = balance - 100 
WHERE account_id = 1;

-- 向账户B增加100元
UPDATE bank_account 
SET balance = balance + 100 
WHERE account_id = 2;

-- 检查账户A余额是否足够（这里简化处理，实际应该先检查）
-- 如果余额不足，可以回滚
-- ROLLBACK;

-- 提交事务
COMMIT;
```

**代码说明**：

- `BEGIN`：开始一个新事务
- `UPDATE bank_account SET balance = balance - 100 WHERE account_id = 1`：从账户A扣除100元
- `UPDATE bank_account SET balance = balance + 100 WHERE account_id = 2`：向账户B增加100元
- `COMMIT`：提交事务，使转账操作生效

**执行结果**：

- 如果两个 UPDATE 都成功，COMMIT 后转账完成
- 账户A余额变为900.00，账户B余额变为600.00
- 如果任何一个操作失败，可以执行 ROLLBACK 撤销整个转账

### 5.6.使用 AUTOCOMMIT 模式

**场景**：关闭自动提交，执行多个操作后统一提交


```sql
-- 关闭自动提交
SET AUTOCOMMIT = 0;

-- 执行多个操作（这些操作不会自动提交）
INSERT INTO product_inventory VALUES (5, '商品E', 250, 75.00);

UPDATE product_inventory 
SET price = price * 1.1 
WHERE product_id = 1;

DELETE FROM product_inventory 
WHERE product_id = 3;

-- 提交所有操作
COMMIT;

-- 重新开启自动提交
SET AUTOCOMMIT = 1;
```

**代码说明**：

- `SET AUTOCOMMIT = 0`：关闭自动提交模式
- `INSERT INTO product_inventory VALUES (5, '商品E', 250, 75.00)`：插入新商品（不会自动提交）
- `UPDATE product_inventory SET price = price * 1.1 WHERE product_id = 1`：更新商品A的价格（不会自动提交）
- `DELETE FROM product_inventory WHERE product_id = 3`：删除商品C（不会自动提交）
- `COMMIT`：提交所有操作，使所有修改生效
- `SET AUTOCOMMIT = 1`：重新开启自动提交模式

**执行结果**：

- 关闭自动提交后，所有 SQL 语句都不会立即提交
- 执行 COMMIT 后，所有操作才会生效
- 如果执行 ROLLBACK，所有操作都会被撤销

## 6.事务的注意事项

### 6.1.事务的自动提交

MySQL 默认情况下，每条 SQL 语句都会自动提交。要使用事务，需要显式开启：


```sql
-- 方式一：使用 BEGIN 或 START TRANSACTION
BEGIN;
-- 执行 SQL 语句
COMMIT;

-- 方式二：关闭自动提交
SET AUTOCOMMIT = 0;
-- 执行 SQL 语句
COMMIT;
SET AUTOCOMMIT = 1;
```

### 6.2.存储引擎要求

只有使用 InnoDB 存储引擎的表才支持事务。创建表时应该指定引擎：


```sql
-- 创建支持事务的表
CREATE TABLE my_table (
    id INT PRIMARY KEY,
    name VARCHAR(50)
) ENGINE=InnoDB;
```

### 6.3.事务中的 DDL 语句

在事务中执行 DDL 语句（如 CREATE、ALTER、DROP）会导致事务自动提交：


```sql
BEGIN;
INSERT INTO my_table VALUES (1, 'test');
-- 执行 DDL 语句会导致事务自动提交
CREATE TABLE new_table (id INT);
-- 此时之前的事务已经提交，无法回滚
ROLLBACK;  -- 这个回滚不会影响已提交的 INSERT
```

**说明**：

- DDL 语句（CREATE、ALTER、DROP、TRUNCATE 等）会自动提交当前事务
- 执行 DDL 后，之前的事务操作已经提交，无法回滚
- 如果需要使用事务，应该避免在事务中执行 DDL 语句

## 7.实际应用示例

### 7.1.创建测试环境

为了更好地演示事务的使用，我们先创建一个完整的测试环境：


```sql
-- 创建学生成绩表
CREATE TABLE student_scores (
    student_id INT PRIMARY KEY,
    student_name VARCHAR(50),
    math_score INT,
    english_score INT,
    total_score INT
) ENGINE=InnoDB;

-- 插入初始数据
INSERT INTO student_scores VALUES
(101, '张三', 85, 90, 175),
(102, '李四', 78, 88, 166),
(103, '王五', 92, 85, 177);
```

### 7.2.示例一：成功提交的事务

**场景**：更新学生的数学成绩，并重新计算总分


```sql
-- 开始事务
START TRANSACTION;

-- 更新张三的数学成绩
UPDATE student_scores 
SET math_score = 90 
WHERE student_id = 101;

-- 重新计算总分
UPDATE student_scores 
SET total_score = math_score + english_score 
WHERE student_id = 101;

-- 查看更新后的结果（在提交前可以查看）
SELECT * FROM student_scores WHERE student_id = 101;

-- 提交事务
COMMIT;
```

**执行流程**：

1. 使用 `START TRANSACTION` 开启事务
2. 执行第一个 UPDATE，将数学成绩改为 90
3. 执行第二个 UPDATE，重新计算总分
4. 使用 `SELECT` 查看结果（此时数据还未提交，只在当前事务中可见）
5. 使用 `COMMIT` 提交事务，使所有修改生效

**结果**：

- 张三的数学成绩更新为 90
- 总分自动更新为 180（90 + 90）
- 所有修改永久保存到数据库

### 7.3.示例二：回滚的事务

**场景**：尝试更新成绩，但发现数据有误，需要回滚


```sql
-- 开始事务
BEGIN;

-- 更新李四的英语成绩
UPDATE student_scores 
SET english_score = 95 
WHERE student_id = 102;

-- 更新李四的总分
UPDATE student_scores 
SET total_score = math_score + english_score 
WHERE student_id = 102;

-- 发现英语成绩输入错误，需要撤销
-- 回滚事务
ROLLBACK;

-- 验证数据是否恢复
SELECT * FROM student_scores WHERE student_id = 102;
```

**执行流程**：

1. 使用 `BEGIN` 开启事务
2. 更新英语成绩和总分
3. 发现数据有误，执行 `ROLLBACK` 回滚
4. 验证数据已恢复到事务开始前的状态

**结果**：

- 所有更新操作被撤销
- 李四的成绩数据恢复到原始状态
- 数据库保持一致性

### 7.4.示例三：使用保存点的复杂事务

**场景**：批量更新多个学生的成绩，但需要部分回滚


```sql
-- 开始事务
START TRANSACTION;

-- 更新张三的成绩
UPDATE student_scores 
SET math_score = 88, 
    total_score = math_score + english_score 
WHERE student_id = 101;

-- 创建保存点 checkpoint1
SAVEPOINT checkpoint1;

-- 更新李四的成绩
UPDATE student_scores 
SET english_score = 92, 
    total_score = math_score + english_score 
WHERE student_id = 102;

-- 创建保存点 checkpoint2
SAVEPOINT checkpoint2;

-- 更新王五的成绩（假设这里出错了）
UPDATE student_scores 
SET math_score = 105  -- 分数超过100，不合理
WHERE student_id = 103;

-- 发现王五的更新有问题，回滚到 checkpoint2
ROLLBACK TO SAVEPOINT checkpoint2;

-- 继续更新王五的成绩（正确的值）
UPDATE student_scores 
SET math_score = 95, 
    total_score = math_score + english_score 
WHERE student_id = 103;

-- 提交事务
COMMIT;
```

**执行流程**：

1. 更新张三的成绩
2. 创建保存点 checkpoint1
3. 更新李四的成绩
4. 创建保存点 checkpoint2
5. 更新王五的成绩（错误的值）
6. 回滚到 checkpoint2，撤销王五的错误更新
7. 重新更新王五的成绩（正确的值）
8. 提交事务

**结果**：

- 张三的成绩更新生效
- 李四的成绩更新生效
- 王五的错误更新被撤销，正确的更新生效
- 只有 checkpoint2 之后到回滚点之间的操作被撤销

### 7.5.示例四：使用 AUTOCOMMIT 模式

**场景**：关闭自动提交，执行多个操作后统一提交或回滚


```sql
-- 查看当前自动提交状态
SELECT @@AUTOCOMMIT;

-- 关闭自动提交
SET AUTOCOMMIT = 0;

-- 执行多个操作（这些操作不会立即提交）
INSERT INTO student_scores VALUES (104, '赵六', 88, 92, 180);

UPDATE student_scores 
SET math_score = math_score + 5 
WHERE student_id = 101;

UPDATE student_scores 
SET english_score = english_score + 3 
WHERE student_id = 102;

-- 查看所有操作的结果
SELECT * FROM student_scores;

-- 如果一切正常，提交所有操作
COMMIT;

-- 或者如果发现问题，回滚所有操作
-- ROLLBACK;

-- 重新开启自动提交
SET AUTOCOMMIT = 1;
```

**执行流程**：

1. 查看当前自动提交状态（默认为 1，即开启）
2. 设置 `AUTOCOMMIT = 0` 关闭自动提交
3. 执行多个 INSERT 和 UPDATE 操作
4. 这些操作不会立即提交，可以统一管理
5. 如果所有操作都正确，执行 `COMMIT` 提交
6. 如果有问题，执行 `ROLLBACK` 回滚
7. 最后重新开启自动提交

**说明**：

- 关闭自动提交后，所有 SQL 语句都在一个隐式事务中
- 需要手动执行 `COMMIT` 或 `ROLLBACK` 来结束事务
- 适合需要批量操作的场景

### 7.6.示例五：模拟订单处理流程

**场景**：创建一个完整的订单处理示例，展示事务在实际业务中的应用

首先创建相关表：


```sql
-- 创建商品表
CREATE TABLE products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(50),
    stock INT,
    price DECIMAL(10, 2)
) ENGINE=InnoDB;

-- 创建订单表
CREATE TABLE orders (
    order_id INT AUTO_INCREMENT PRIMARY KEY,
    customer_name VARCHAR(50),
    order_date DATETIME,
    total_amount DECIMAL(10, 2)
) ENGINE=InnoDB;

-- 创建订单明细表
CREATE TABLE order_items (
    item_id INT AUTO_INCREMENT PRIMARY KEY,
    order_id INT,
    product_id INT,
    quantity INT,
    subtotal DECIMAL(10, 2),
    FOREIGN KEY (order_id) REFERENCES orders(order_id)
) ENGINE=InnoDB;

-- 插入商品数据
INSERT INTO products VALUES
(1, '笔记本电脑', 50, 5999.00),
(2, '鼠标', 200, 99.00),
(3, '键盘', 150, 299.00);
```

**订单处理事务**：


```sql
-- 开始事务
START TRANSACTION;

-- 1. 创建订单
INSERT INTO orders (customer_name, order_date, total_amount) 
VALUES ('客户A', NOW(), 0);

-- 获取刚插入的订单ID（在同一个事务中）
SET @order_id = LAST_INSERT_ID();

-- 2. 添加订单明细：购买2台笔记本电脑
INSERT INTO order_items (order_id, product_id, quantity, subtotal)
VALUES (@order_id, 1, 2, 5999.00 * 2);

-- 3. 更新商品库存
UPDATE products 
SET stock = stock - 2 
WHERE product_id = 1;

-- 4. 添加订单明细：购买5个鼠标
INSERT INTO order_items (order_id, product_id, quantity, subtotal)
VALUES (@order_id, 2, 5, 99.00 * 5);

-- 5. 更新商品库存
UPDATE products 
SET stock = stock - 5 
WHERE product_id = 2;

-- 6. 计算并更新订单总金额
UPDATE orders 
SET total_amount = (
    SELECT SUM(subtotal) 
    FROM order_items 
    WHERE order_id = @order_id
)
WHERE order_id = @order_id;

-- 检查库存是否充足（这里简化处理）
-- 如果库存不足，可以执行 ROLLBACK

-- 提交事务
COMMIT;
```

**代码说明**：

- 整个订单处理流程在一个事务中完成
- 包括创建订单、添加订单明细、更新库存、计算总金额等多个步骤
- 如果任何一步失败，整个事务都会回滚
- 保证订单数据和库存数据的一致性

**执行结果**：

- 如果所有操作成功，订单创建完成，库存正确更新
- 如果任何操作失败，整个订单处理流程回滚，数据保持一致

## 8.事务的自动提交机制

### 8.1.默认自动提交行为

MySQL 默认情况下，每条 SQL 语句执行后都会自动提交。这意味着：


```sql
-- 默认情况下，每条语句都会自动提交
INSERT INTO student_scores VALUES (105, '孙七', 90, 85, 175);
-- 这条 INSERT 语句执行后立即提交，无法回滚

UPDATE student_scores SET math_score = 95 WHERE student_id = 101;
-- 这条 UPDATE 语句执行后立即提交，无法回滚
```

### 8.2.显式开启事务

要使用事务功能，需要显式开启事务：


```sql
-- 方式一：使用 BEGIN
BEGIN;
-- 执行 SQL 语句
COMMIT;

-- 方式二：使用 START TRANSACTION
START TRANSACTION;
-- 执行 SQL 语句
COMMIT;

-- 方式三：关闭自动提交
SET AUTOCOMMIT = 0;
-- 执行 SQL 语句
COMMIT;
SET AUTOCOMMIT = 1;
```

### 8.3.自动提交的影响

**重要提示**：

- 在 MySQL 命令行中，默认情况下事务是自动提交的
- 执行 SQL 语句后就会立即执行 COMMIT 操作
- 要使用事务，必须显式使用 `BEGIN` 或 `START TRANSACTION` 开启事务
- 或者使用 `SET AUTOCOMMIT=0` 关闭当前会话的自动提交

## 9.事务处理的最佳实践

### 9.1.合理使用事务

**何时使用事务**：

- 需要保证多个操作要么全部成功，要么全部失败
- 涉及多个表的关联操作
- 需要保证数据一致性的场景

**何时不使用事务**：

- 简单的单表查询操作
- 不需要保证一致性的独立操作
- 对性能要求极高的场景（事务会有一定的性能开销）

### 9.2.事务的范围

**原则**：

- 事务应该尽可能短，减少锁定时间
- 不要在事务中执行耗时的操作
- 不要在事务中进行用户交互

**示例**：


```sql
-- 好的做法：事务范围小，操作快速
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- 不好的做法：事务范围大，包含耗时操作
BEGIN;
-- 执行大量数据处理
-- 等待用户输入（不应该在事务中）
-- 提交事务
COMMIT;
```

### 9.3.错误处理

**原则**：

- 在事务中应该检查每个操作的结果
- 如果发现错误，应该立即回滚
- 不要忽略错误，让事务继续执行

**示例**：


```sql
-- 好的做法：检查错误并回滚
START TRANSACTION;

UPDATE user_account SET balance = balance - 200 WHERE user_id = 1;

-- 检查余额是否足够（这里简化，实际应该先查询）
-- 如果余额不足，执行回滚
-- IF (余额不足) THEN
--     ROLLBACK;
-- ELSE
    UPDATE user_account SET balance = balance + 200 WHERE user_id = 2;
    COMMIT;
-- END IF;
```

### 9.4.保存点的使用

**原则**：

- 在复杂事务中合理使用保存点
- 保存点可以帮助部分回滚，提高灵活性
- 但不要创建过多的保存点，会增加复杂度

**示例**：


```sql
START TRANSACTION;

-- 执行操作A
UPDATE table1 SET column1 = 'value1';

-- 创建保存点
SAVEPOINT point1;

-- 执行操作B
UPDATE table2 SET column2 = 'value2';

-- 如果操作B有问题，可以回滚到point1
-- ROLLBACK TO SAVEPOINT point1;

COMMIT;
```

## 10.常见问题与解答

### 10.1.事务会自动提交吗？

**答**：MySQL 默认情况下，每条 SQL 语句都会自动提交。要使用事务，需要：

- 使用 `BEGIN` 或 `START TRANSACTION` 显式开启事务
- 或者使用 `SET AUTOCOMMIT = 0` 关闭自动提交

### 10.2.如何查看当前是否在事务中？

**答**：可以使用以下方法：


```sql
-- 查看自动提交状态
SELECT @@AUTOCOMMIT;

-- 查看当前事务状态（在某些 MySQL 版本中）
SHOW VARIABLES LIKE 'autocommit';
```

### 10.3.事务中可以执行哪些操作？

**答**：

- **可以执行**：INSERT、UPDATE、DELETE、SELECT 等 DML 语句
- **需要注意**：CREATE、ALTER、DROP 等 DDL 语句会导致事务自动提交

### 10.4.事务回滚后，自增 ID 会恢复吗？

**答**：不会。自增 ID 一旦分配就不会回收，即使事务回滚，自增 ID 也不会恢复。这是 MySQL 的设计，目的是保证 ID 的唯一性。

**示例**：


```sql
CREATE TABLE test_table (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50)
) ENGINE=InnoDB;

BEGIN;
INSERT INTO test_table (name) VALUES ('test1');  -- 假设 id = 1
INSERT INTO test_table (name) VALUES ('test2');  -- 假设 id = 2
ROLLBACK;

-- 再次插入
INSERT INTO test_table (name) VALUES ('test3');  -- id = 3（不是 1）
```

### 10.5.多个事务可以同时执行吗？

**答**：可以。MySQL 支持多个事务并发执行，通过隔离级别来控制事务之间的可见性。不同的隔离级别会影响并发行为。

### 10.6.如何选择合适的隔离级别？

**答**：

- **读未提交（READ UNCOMMITTED）**：性能最好，但数据一致性最差，一般不推荐
- **读已提交（READ COMMITTED）**：平衡性能和一致性，适合大多数场景
- **可重复读（REPEATABLE READ）**：MySQL 默认级别，保证同一事务中读取结果一致
- **串行化（SERIALIZABLE）**：数据一致性最好，但性能最差，适合对一致性要求极高的场景
