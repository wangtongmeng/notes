## 1.引言

在 MySQL 中，数据类型定义了数据字段可以存储什么类型的数据以及数据的格式。正确选择数据类型对于数据库的性能优化和存储效率至关重要。

本文介绍 MySQL 支持的主要数据类型，包括数值类型、日期时间类型、字符串类型、枚举集合类型和空间数据类型。通过实际示例，你将学会如何根据实际需求选择合适的数据类型。

## 2.数据类型基础概念

### 2.1.什么是数据类型？

数据类型（Data Type）规定了数据在数据库中的存储格式、取值范围和可执行的操作。就像在编程语言中，我们需要声明变量类型一样，在创建数据库表时，也需要为每个字段指定数据类型。

**为什么数据类型很重要？**

- **存储效率**：选择合适的数据类型可以节省存储空间
- **性能优化**：正确的数据类型有助于提高查询和索引效率
- **数据完整性**：数据类型可以限制数据的格式，确保数据正确性
- **功能支持**：某些数据类型支持特定的操作和函数

### 2.2.MySQL 数据类型分类

MySQL 支持的数据类型大致可以分为以下几类：

- **数值类型**：用于存储数字，包括整数和浮点数
- **日期时间类型**：用于存储日期和时间信息
- **字符串类型**：用于存储文本和字符数据
- **枚举集合类型**：用于存储预定义的选项值
- **空间数据类型**：用于存储地理信息和几何图形

## 3.数值类型

### 3.1.整数类型

整数类型用于存储整数值。MySQL 提供了多种整数类型，每种类型占用的存储空间和取值范围不同。

#### 3.1.1.TINYINT

`TINYINT` 是最小的整数类型，占用 1 个字节。

**存储范围**：

- 有符号（SIGNED）：-128 到 127
- 无符号（UNSIGNED）：0 到 255

**适用场景**：

- 存储年龄（0-150）
- 存储状态码（0-255）
- 存储百分比（0-100）

**示例**：


```sql
CREATE TABLE user_info (
    id INT PRIMARY KEY,
    age TINYINT UNSIGNED,
    status TINYINT DEFAULT 1
);
```

#### 3.1.2.SMALLINT

`SMALLINT` 占用 2 个字节，适合存储中等范围的整数。

**存储范围**：

- 有符号：-32,768 到 32,767
- 无符号：0 到 65,535

**适用场景**：

- 存储年份（如 1900-2100）
- 存储端口号（0-65535）
- 存储较小的计数

**示例**：


```sql
CREATE TABLE server_config (
    id INT PRIMARY KEY,
    port SMALLINT UNSIGNED,
    year_founded SMALLINT
);
```

#### 3.1.3.MEDIUMINT

`MEDIUMINT` 占用 3 个字节，介于 SMALLINT 和 INT 之间。

**存储范围**：

- 有符号：-8,388,608 到 8,388,607
- 无符号：0 到 16,777,215

**适用场景**：

- 存储较大的计数或ID
- 存储颜色值（RGB，0-16777215）

**示例**：


```sql
CREATE TABLE product (
    id INT PRIMARY KEY,
    color_code MEDIUMINT UNSIGNED,
    view_count MEDIUMINT UNSIGNED DEFAULT 0
);
```

#### 3.1.4.INT 或 INTEGER

`INT`（或 `INTEGER`）是最常用的整数类型，占用 4 个字节。关键字 `INT` 和 `INTEGER` 是同义词。

**存储范围**：

- 有符号：-2,147,483,648 到 2,147,483,647
- 无符号：0 到 4,294,967,295

**适用场景**：

- 主键 ID（通常使用 `INT UNSIGNED` 或 `BIGINT`）
- 存储用户数量、订单数量等
- 大多数需要存储整数的场景

**示例**：


```sql
CREATE TABLE orders (
    order_id INT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
    user_id INT UNSIGNED,
    total_amount INT
);
```

#### 3.1.5.BIGINT

`BIGINT` 是最大的整数类型，占用 8 个字节。

**存储范围**：

- 有符号：-9,223,372,036,854,775,808 到 9,223,372,036,854,775,807
- 无符号：0 到 18,446,744,073,709,551,615

**适用场景**：

- 大型系统的主键 ID
- 存储时间戳（毫秒级）
- 存储非常大的计数

**示例**：


```sql
CREATE TABLE transaction_log (
    log_id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
    timestamp_ms BIGINT UNSIGNED,
    amount BIGINT
);
```

### 3.2.浮点数类型

浮点数类型用于存储带小数点的数值。

#### 3.2.1.FLOAT

`FLOAT` 是单精度浮点数，占用 4 个字节。

**存储范围**：

- 有符号：约 -3.4×10³⁸ 到 3.4×10³⁸
- 无符号：0 到约 3.4×10³⁸
- 精度：约 7 位有效数字

**适用场景**：

- 存储不需要高精度的浮点数
- 科学计算中的近似值
- 存储温度、重量等测量值

**示例**：


```sql
CREATE TABLE sensor_data (
    id INT PRIMARY KEY,
    temperature FLOAT,
    humidity FLOAT
);
```

#### 3.2.2.DOUBLE

`DOUBLE` 是双精度浮点数，占用 8 个字节。

**存储范围**：

- 有符号：约 -1.8×10³⁰⁸ 到 1.8×10³⁰⁸
- 无符号：0 到约 1.8×10³⁰⁸
- 精度：约 15 位有效数字

**适用场景**：

- 需要更高精度的浮点数计算
- 科学计算和工程应用
- 金融计算（但通常使用 DECIMAL 更合适）

**示例**：


```sql
CREATE TABLE calculation_result (
    id INT PRIMARY KEY,
    result_value DOUBLE,
    precision_data DOUBLE
);
```

### 3.3.定点数类型

#### 3.3.1.DECIMAL

`DECIMAL`（或 `DEC`）是定点数类型，用于存储精确的小数值。存储空间取决于精度定义。

**语法**：


```sql
DECIMAL(M, D)
```

**参数说明**：

- `M`：总位数（精度），范围 1-65
- `D`：小数位数（标度），范围 0-30，且 D ≤ M
- 存储空间：如果 M > D，为 M+2 字节；否则为 D+2 字节

**适用场景**：

- 货币金额（需要精确计算）
- 需要精确小数的场景
- 避免浮点数精度问题

**示例**：


```sql
CREATE TABLE payment (
    id INT PRIMARY KEY,
    amount DECIMAL(10, 2),  -- 总共10位，小数点后2位，如 99999999.99
    tax_rate DECIMAL(5, 4)  -- 总共5位，小数点后4位，如 0.1234
);
```

### 3.4.BIT 类型

`BIT` 类型用于存储位字段值，支持 MyISAM、MEMORY、InnoDB 和 BDB 表。

**语法**：


```sql
BIT(M)
```

**参数说明**：

- `M`：位数，范围 1-64，默认为 1

**适用场景**：

- 存储布尔值（BIT(1)）
- 存储标志位组合
- 位操作场景

**示例**：


```sql
CREATE TABLE user_settings (
    id INT PRIMARY KEY,
    is_active BIT(1) DEFAULT b'1',
    flags BIT(8)
);
```

### 3.5.数值类型选择建议

选择数值类型时，考虑以下因素：

1. **取值范围**：选择能够容纳所有可能值的最小类型
2. **存储空间**：在满足需求的前提下，选择占用空间较小的类型
3. **精度要求**：需要精确计算时使用 DECIMAL，可以接受近似值时使用 FLOAT 或 DOUBLE
4. **性能考虑**：整数类型通常比浮点数类型性能更好

**数值类型对比表**：

| 类型         | 大小（字节） | 有符号范围                      | 无符号范围         | 主要用途     |
| ------------ | ------------ | ------------------------------- | ------------------ | ------------ |
| TINYINT      | 1            | -128 到 127                     | 0 到 255           | 小整数值     |
| SMALLINT     | 2            | -32,768 到 32,767               | 0 到 65,535        | 中等整数值   |
| MEDIUMINT    | 3            | -8,388,608 到 8,388,607         | 0 到 16,777,215    | 较大整数值   |
| INT/INTEGER  | 4            | -2,147,483,648 到 2,147,483,647 | 0 到 4,294,967,295 | 标准整数值   |
| BIGINT       | 8            | -9.22×10¹⁸ 到 9.22×10¹⁸         | 0 到 1.84×10¹⁹     | 极大整数值   |
| FLOAT        | 4            | ±3.4×10³⁸                       | 0 到 3.4×10³⁸      | 单精度浮点数 |
| DOUBLE       | 8            | ±1.8×10³⁰⁸                      | 0 到 1.8×10³⁰⁸     | 双精度浮点数 |
| DECIMAL(M,D) | 可变         | 依赖于 M 和 D                   | 依赖于 M 和 D      | 精确小数值   |

## 4.日期和时间类型

### 4.1.DATE

`DATE` 类型用于存储日期值，不包含时间信息。

**存储范围**：1000-01-01 到 9999-12-31

**格式**：YYYY-MM-DD

**存储空间**：3 字节

**适用场景**：

- 存储生日
- 存储事件日期
- 存储截止日期

**示例**：


```sql
CREATE TABLE employee (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    birth_date DATE,
    hire_date DATE
);

-- 插入数据
INSERT INTO employee VALUES (1, '张三', '1990-05-15', '2020-03-01');
```

### 4.2.TIME

`TIME` 类型用于存储时间值或持续时间。

**存储范围**：'-838:59:59' 到 '838:59:59'

**格式**：HH:MM:SS 或 HHH:MM:SS（用于持续时间）

**存储空间**：3 字节

**适用场景**：

- 存储工作时间
- 存储持续时间
- 存储时间间隔

**示例**：


```sql
CREATE TABLE work_schedule (
    id INT PRIMARY KEY,
    start_time TIME,
    end_time TIME,
    duration TIME
);

-- 插入数据
INSERT INTO work_schedule VALUES (1, '09:00:00', '18:00:00', '09:00:00');
```

### 4.3.YEAR

`YEAR` 类型用于存储年份值。

**存储范围**：1901 到 2155

**格式**：YYYY（4 位）或 YY（2 位，70-99 表示 1970-1999，00-69 表示 2000-2069）

**存储空间**：1 字节

**适用场景**：

- 存储年份信息
- 存储生产年份
- 存储毕业年份

**示例**：


```sql
CREATE TABLE car (
    id INT PRIMARY KEY,
    brand VARCHAR(50),
    model_year YEAR
);

-- 插入数据
INSERT INTO car VALUES (1, 'Toyota', 2023);
INSERT INTO car VALUES (2, 'Honda', '24');  -- 表示 2024
```

### 4.4.DATETIME

`DATETIME` 类型用于存储日期和时间的组合值。

**存储范围**：'1000-01-01 00:00:00' 到 '9999-12-31 23:59:59'

**格式**：YYYY-MM-DD HH:MM:SS

**存储空间**：8 字节

**适用场景**：

- 存储创建时间
- 存储更新时间
- 存储事件发生时间
- 需要存储完整日期时间信息的场景

**示例**：


```sql
CREATE TABLE article (
    id INT PRIMARY KEY,
    title VARCHAR(200),
    created_at DATETIME,
    updated_at DATETIME
);

-- 插入数据
INSERT INTO article VALUES (1, 'MySQL 教程', '2024-01-15 10:30:00', '2024-01-20 14:20:00');
```

### 4.5.TIMESTAMP

`TIMESTAMP` 类型也用于存储日期和时间，但有一些特殊特性。

**存储范围**：

- '1970-01-01 00:00:01' UTC 到 '2038-01-19 03:14:07' UTC
- 结束时间对应北京时间：2038-01-19 11:14:07

**格式**：YYYY-MM-DD HH:MM:SS

**存储空间**：4 字节

**特殊特性**：

- 自动时区转换：存储时转换为 UTC，检索时转换回当前时区
- 自动更新：可以设置为在插入或更新时自动更新为当前时间

**适用场景**：

- 记录最后修改时间（利用自动更新特性）
- 需要时区转换的场景
- 存储时间戳（但注意 2038 年限制）

**示例**：


```sql
CREATE TABLE user_login (
    id INT PRIMARY KEY,
    username VARCHAR(50),
    login_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_activity TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 插入数据时，login_time 和 last_activity 会自动设置为当前时间
INSERT INTO user_login (username) VALUES ('admin');

-- 更新数据时，last_activity 会自动更新为当前时间
UPDATE user_login SET username = 'administrator' WHERE id = 1;
```

### 4.6.日期时间类型选择建议

**DATE vs DATETIME vs TIMESTAMP**：

- **DATE**：只需要日期，不需要时间
- **DATETIME**：需要完整的日期时间，范围更大，不受时区影响
- **TIMESTAMP**：需要日期时间，且需要自动更新或时区转换功能

**日期时间类型对比表**：

| 类型      | 大小（字节） | 范围                                                   | 格式                | 主要用途                 |
| --------- | ------------ | ------------------------------------------------------ | ------------------- | ------------------------ |
| DATE      | 3            | 1000-01-01 到 9999-12-31                               | YYYY-MM-DD          | 日期值                   |
| TIME      | 3            | '-838:59:59' 到 '838:59:59'                            | HH:MM:SS            | 时间值或持续时间         |
| YEAR      | 1            | 1901 到 2155                                           | YYYY                | 年份值                   |
| DATETIME  | 8            | '1000-01-01 00:00:00' 到 '9999-12-31 23:59:59'         | YYYY-MM-DD HH:MM:SS | 混合日期和时间值         |
| TIMESTAMP | 4            | '1970-01-01 00:00:01' UTC 到 '2038-01-19 03:14:07' UTC | YYYY-MM-DD HH:MM:SS | 混合日期和时间值，时间戳 |

## 5.字符串类型

### 5.1.定长字符串类型

#### 5.1.1.CHAR

`CHAR` 是定长字符串类型，存储固定长度的字符串。

**语法**：


```sql
CHAR(n)
```

**参数说明**：

- `n`：字符个数，范围 0-255
- 存储空间：固定为 n 字节（对于单字节字符集）

**特点**：

- 如果存储的字符串长度小于 n，会在右侧用空格填充
- 检索时会自动去除尾部空格
- 适合存储长度固定的字符串

**适用场景**：

- 存储固定长度的代码（如国家代码、状态码）
- 存储固定格式的标识符
- 长度变化不大的短字符串

**示例**：


```sql
CREATE TABLE country (
    id INT PRIMARY KEY,
    code CHAR(2),        -- 国家代码，如 'CN', 'US'
    name VARCHAR(100)
);

INSERT INTO country VALUES (1, 'CN', '中国');
INSERT INTO country VALUES (2, 'US', '美国');
```

### 5.2.变长字符串类型

#### 5.2.1.VARCHAR

`VARCHAR` 是变长字符串类型，存储可变长度的字符串。

**语法**：


```sql
VARCHAR(n)
```

**参数说明**：

- `n`：最大字符个数，范围 0-65,535
- 存储空间：实际长度 + 1 或 2 字节（用于存储长度信息）

**特点**：

- 只存储实际使用的字符，不填充空格
- 尾部空格会被保留
- 适合存储长度变化较大的字符串

**适用场景**：

- 存储用户名、姓名等
- 存储地址、描述等文本
- 大多数需要存储文本的场景

**示例**：


```sql
CREATE TABLE customer (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    email VARCHAR(100),
    address VARCHAR(200)
);

INSERT INTO customer VALUES (1, '李四', 'lisi@example.com', '北京市朝阳区');
```

### 5.3.CHAR 与 VARCHAR 的区别

**存储方式**：

- `CHAR`：固定长度，总是占用 n 字节
- `VARCHAR`：可变长度，只占用实际使用的空间

**性能考虑**：

- `CHAR`：对于固定长度的数据，性能略好
- `VARCHAR`：对于长度变化大的数据，节省存储空间

**选择建议**：

- 长度固定或变化很小：使用 `CHAR`
- 长度变化较大：使用 `VARCHAR`

### 5.4.二进制字符串类型

#### 5.4.1.BINARY 和 VARBINARY

`BINARY` 和 `VARBINARY` 类似于 `CHAR` 和 `VARCHAR`，但存储的是二进制字符串（字节字符串）而不是字符字符串。

**特点**：

- 没有字符集概念
- 排序和比较基于字节值
- 区分大小写

**适用场景**：

- 存储二进制数据
- 需要区分大小写的场景
- 存储加密数据

**示例**：


```sql
CREATE TABLE secure_data (
    id INT PRIMARY KEY,
    token BINARY(32),
    hash VARBINARY(64)
);
```

### 5.5.文本类型

文本类型用于存储较长的文本数据。

#### 5.5.1.TINYTEXT

`TINYTEXT` 用于存储短文本字符串。

**存储范围**：0-255 字节

**适用场景**：

- 存储简短的描述
- 存储备注信息

**示例**：


```sql
CREATE TABLE product (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    short_desc TINYTEXT
);
```

#### 5.5.2.TEXT

`TEXT` 用于存储中等长度的文本数据。

**存储范围**：0-65,535 字节（约 64 KB）

**适用场景**：

- 存储文章内容
- 存储产品描述
- 存储评论内容

**示例**：


```sql
CREATE TABLE blog_post (
    id INT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT
);
```

#### 5.5.3.MEDIUMTEXT

`MEDIUMTEXT` 用于存储中等长度的文本数据。

**存储范围**：0-16,777,215 字节（约 16 MB）

**适用场景**：

- 存储较长的文章
- 存储详细的文档内容

**示例**：


```sql
CREATE TABLE document (
    id INT PRIMARY KEY,
    title VARCHAR(200),
    body MEDIUMTEXT
);
```

#### 5.5.4.LONGTEXT

`LONGTEXT` 用于存储极大的文本数据。

**存储范围**：0-4,294,967,295 字节（约 4 GB）

**适用场景**：

- 存储非常大的文档
- 存储完整的网页内容
- 存储日志文件内容

**示例**：


```sql
CREATE TABLE archive (
    id INT PRIMARY KEY,
    filename VARCHAR(255),
    content LONGTEXT
);
```

### 5.6.二进制大对象类型（BLOB）

BLOB（Binary Large Object）类型用于存储二进制形式的大对象数据。

#### 5.6.1.TINYBLOB

**存储范围**：0-255 字节

**适用场景**：存储小的二进制数据

#### 5.6.2.BLOB

`BLOB` 用于存储中等长度的二进制数据。

**存储范围**：0-65,535 字节（约 64 KB）

**适用场景**：

- 存储图片文件（小到中等大小）
- 存储文档文件
- 存储序列化的数据

**示例**：


```sql
CREATE TABLE image_storage (
    id INT PRIMARY KEY,
    filename VARCHAR(255),
    image_data BLOB
);
```

#### 5.6.3.MEDIUMBLOB

`MEDIUMBLOB` 用于存储较大长度的二进制数据。

**存储范围**：0-16,777,215 字节（约 16 MB）

**适用场景**：

- 存储较大的图片文件
- 存储视频文件片段
- 存储压缩文件

**示例**：


```sql
CREATE TABLE media_file (
    id INT PRIMARY KEY,
    file_name VARCHAR(255),
    file_content MEDIUMBLOB
);
```

#### 5.6.4.LONGBLOB

`LONGBLOB` 用于存储极大的二进制数据。

**存储范围**：0-4,294,967,295 字节（约 4 GB）

**适用场景**：

- 存储非常大的文件
- 存储完整的视频文件
- 存储大型数据库备份

**示例**：


```sql
CREATE TABLE large_file (
    id INT PRIMARY KEY,
    file_name VARCHAR(255),
    file_data LONGBLOB
);
```

### 5.7.TEXT 与 BLOB 的区别

**TEXT 类型**：

- 存储文本数据（字符数据）
- 有字符集概念
- 可以按字符集进行排序和比较

**BLOB 类型**：

- 存储二进制数据（字节数据）
- 没有字符集概念
- 按字节值进行排序和比较

**选择建议**：

- 存储文本内容：使用 TEXT 系列
- 存储二进制文件：使用 BLOB 系列

### 5.8.字符串类型选择建议

选择字符串类型时，考虑以下因素：

1. **数据长度**：
   - 固定长度或很短：使用 `CHAR`
   - 长度变化：使用 `VARCHAR`
   - 超过 255 字符：使用 `TEXT` 系列
2. **数据类型**：
   - 文本数据：使用 `CHAR`、`VARCHAR`、`TEXT` 系列
   - 二进制数据：使用 `BINARY`、`VARBINARY`、`BLOB` 系列
3. **存储空间**：
   - 固定长度数据：`CHAR` 可能更高效
   - 变长数据：`VARCHAR` 节省空间

**字符串类型对比表**：

| 类型       | 大小                 | 主要用途                 |
| ---------- | -------------------- | ------------------------ |
| CHAR(n)    | 0-255 字节           | 定长字符串               |
| VARCHAR(n) | 0-65,535 字节        | 变长字符串               |
| TINYTEXT   | 0-255 字节           | 短文本字符串             |
| TEXT       | 0-65,535 字节        | 长文本数据               |
| MEDIUMTEXT | 0-16,777,215 字节    | 中等长度文本数据         |
| LONGTEXT   | 0-4,294,967,295 字节 | 极大文本数据             |
| TINYBLOB   | 0-255 字节           | 小二进制数据             |
| BLOB       | 0-65,535 字节        | 二进制形式的长文本数据   |
| MEDIUMBLOB | 0-16,777,215 字节    | 二进制形式的中等长度数据 |
| LONGBLOB   | 0-4,294,967,295 字节 | 二进制形式的极大数据     |

**重要提示**：

- `CHAR(n)` 和 `VARCHAR(n)` 中的 `n` 代表字符个数，不是字节个数
- 例如 `CHAR(30)` 可以存储 30 个字符，对于 UTF-8 编码，可能需要 30-90 字节
- 选择长度时，要考虑字符集的影响

## 6.枚举与集合类型

### 6.1.ENUM 类型

`ENUM`（枚举）类型用于存储单一值，该值必须从预定义的选项列表中选择。

**语法**：


```sql
ENUM('值1', '值2', '值3', ...)
```

**特点**：

- 只能存储列表中的一个值
- 在内部以整数形式存储，节省空间
- 最多可以定义 65,535 个不同的值

**适用场景**：

- 存储状态值（如：'pending', 'approved', 'rejected'）
- 存储类型值（如：'admin', 'user', 'guest'）
- 存储固定的选项值

**示例**：


```sql
CREATE TABLE task (
    id INT PRIMARY KEY AUTO_INCREMENT,
    title VARCHAR(200),
    status ENUM('待处理', '进行中', '已完成', '已取消') DEFAULT '待处理',
    priority ENUM('低', '中', '高') DEFAULT '中'
);

-- 插入数据
INSERT INTO task (title, status, priority) 
VALUES ('完成报告', '进行中', '高');

-- 查询特定状态的任务
SELECT * FROM task WHERE status = '已完成';
```

**注意事项**：

- 如果插入的值不在枚举列表中，MySQL 会插入空字符串（如果允许 NULL）或报错
- 枚举值区分大小写
- 枚举值按定义顺序排序，而不是按字母顺序

### 6.2.SET 类型

`SET`（集合）类型用于存储多个值，这些值必须从预定义的选项列表中选择。

**语法**：


```sql
SET('值1', '值2', '值3', ...)
```

**特点**：

- 可以存储列表中的多个值（用逗号分隔）
- 在内部以位图形式存储，非常节省空间
- 最多可以定义 64 个不同的值

**适用场景**：

- 存储多个标签（如：'技术', '设计', '产品'）
- 存储权限组合（如：'read', 'write', 'delete'）
- 存储多选选项

**示例**：


```sql
CREATE TABLE article (
    id INT PRIMARY KEY AUTO_INCREMENT,
    title VARCHAR(200),
    tags SET('技术', '设计', '产品', '运营', '市场') DEFAULT '',
    permissions SET('read', 'write', 'delete', 'admin') DEFAULT 'read'
);

-- 插入数据（可以包含多个值）
INSERT INTO article (title, tags, permissions) 
VALUES ('MySQL 教程', '技术,数据库', 'read,write');

-- 查询包含特定标签的文章
SELECT * FROM article WHERE FIND_IN_SET('技术', tags) > 0;

-- 查询具有特定权限的文章
SELECT * FROM article WHERE permissions LIKE '%write%';
```

**注意事项**：

- 多个值之间用逗号分隔，但不能有空格
- 如果插入的值不在集合列表中，会被忽略
- 可以使用 `FIND_IN_SET()` 函数查询包含特定值的记录

### 6.3.ENUM 与 SET 的区别

**ENUM**：

- 只能存储一个值
- 适合单选场景
- 内部存储更简单

**SET**：

- 可以存储多个值
- 适合多选场景
- 内部以位图存储，更节省空间

**选择建议**：

- 只需要一个选项：使用 `ENUM`
- 需要多个选项：使用 `SET`

### 6.4.枚举集合类型使用建议

1. **选项数量**：
   - `ENUM` 最多 65,535 个选项
   - `SET` 最多 64 个选项
2. **性能考虑**：
   - 枚举和集合类型在内部以整数或位图存储，查询性能较好
   - 适合用于经常查询和过滤的字段
3. **灵活性**：
   - 如果需要经常添加新选项，考虑使用 `VARCHAR` 或单独的关系表
   - 枚举和集合类型修改选项列表需要修改表结构

## 7.空间数据类型

### 7.1.什么是空间数据类型？

空间数据类型用于存储地理信息和几何图形数据，如点、线、面等。这些类型基于 OpenGIS 规范实现。

**主要类型**：

- `GEOMETRY`：通用几何类型，可以存储任何几何对象
- `POINT`：点
- `LINESTRING`：线
- `POLYGON`：多边形
- `MULTIPOINT`：多点
- `MULTILINESTRING`：多线
- `MULTIPOLYGON`：多多边形
- `GEOMETRYCOLLECTION`：几何集合

### 7.2.POINT 类型

`POINT` 用于存储一个坐标点，通常表示经度和纬度。

**语法**：


```sql
POINT(x, y)
```

**适用场景**：

- 存储地理位置坐标
- 存储地图上的标记点
- GPS 定位数据

**示例**：


```sql
CREATE TABLE location (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    coordinates POINT NOT NULL,
    SPATIAL INDEX idx_coordinates (coordinates)
);

-- 插入数据（使用 ST_GeomFromText 函数）
INSERT INTO location (name, coordinates) 
VALUES ('北京天安门', ST_GeomFromText('POINT(116.3974 39.9093)'));

-- 查询数据
SELECT id, name, ST_AsText(coordinates) AS location 
FROM location;
```

### 7.3.LINESTRING 类型

`LINESTRING` 用于存储一条线段，由多个点连接而成。

**语法**：


```sql
LINESTRING(point1, point2, point3, ...)
```

**适用场景**：

- 存储路径（如路线、边界）
- 存储轨迹数据

**示例**：


```sql
CREATE TABLE route (
    id INT PRIMARY KEY AUTO_INCREMENT,
    route_name VARCHAR(100),
    path LINESTRING NOT NULL,
    SPATIAL INDEX idx_path (path)
);

-- 插入数据
INSERT INTO route (route_name, path) 
VALUES ('北京到上海', ST_GeomFromText('LINESTRING(116.3974 39.9093, 121.4737 31.2304)'));
```

### 7.4.POLYGON 类型

`POLYGON` 用于存储一个多边形区域，由闭合的线段组成。

**语法**：


```sql
POLYGON((point1, point2, point3, ..., point1))
```

**适用场景**：

- 存储区域边界
- 存储服务范围
- 存储地理区域

**示例**：


```sql
CREATE TABLE region (
    id INT PRIMARY KEY AUTO_INCREMENT,
    region_name VARCHAR(100),
    boundary POLYGON NOT NULL,
    SPATIAL INDEX idx_boundary (boundary)
);

-- 插入数据（注意：多边形必须闭合，第一个点和最后一个点相同）
INSERT INTO region (region_name, boundary) 
VALUES ('北京市', ST_GeomFromText('POLYGON((116.0 39.0, 117.0 39.0, 117.0 40.0, 116.0 40.0, 116.0 39.0))'));
```

### 7.5.空间数据类型使用建议

1. **存储引擎**：
   - 空间数据类型需要 MyISAM 或 InnoDB 存储引擎（MySQL 5.7.5+）
   - 建议使用 InnoDB 以获得更好的事务支持
2. **索引**：
   - 使用 `SPATIAL INDEX` 创建空间索引
   - 空间索引可以提高空间查询的性能
3. **函数**：
   - 使用 `ST_GeomFromText()` 从文本创建几何对象
   - 使用 `ST_AsText()` 将几何对象转换为文本
   - 使用 `ST_Distance()` 计算两点间距离
   - 使用 `ST_Contains()` 判断是否包含
4. **适用场景**：
   - 地理信息系统（GIS）
   - 位置服务应用
   - 地图应用
   - 物流和配送系统

**注意**：空间数据类型是相对高级的功能，如果只是存储简单的经纬度，使用 `DECIMAL` 或 `DOUBLE` 类型可能更简单直接。

## 8.数据类型选择实践建议

### 8.1.选择数据类型的原则

在实际项目中，选择数据类型时应该遵循以下原则：

1. **够用即可**：选择能够满足需求的最小类型，节省存储空间
2. **考虑扩展**：预留一定的扩展空间，但不要过度预留
3. **性能优先**：对于频繁查询的字段，选择性能更好的类型
4. **精确性**：需要精确计算的场景（如金额），使用 `DECIMAL` 而不是 `FLOAT` 或 `DOUBLE`

### 8.2.常见场景的数据类型选择

**用户表**：


```sql
CREATE TABLE users (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,  -- 大型系统使用 BIGINT
    username VARCHAR(50) NOT NULL UNIQUE,           -- 用户名
    email VARCHAR(100) NOT NULL UNIQUE,             -- 邮箱
    password_hash CHAR(64),                         -- 密码哈希（固定长度）
    age TINYINT UNSIGNED,                           -- 年龄
    status ENUM('active', 'inactive', 'banned') DEFAULT 'active',  -- 状态
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,  -- 创建时间
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP  -- 更新时间
);
```

**订单表**：


```sql
CREATE TABLE orders (
    order_id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT UNSIGNED NOT NULL,
    total_amount DECIMAL(10, 2) NOT NULL,          -- 金额使用 DECIMAL
    status ENUM('pending', 'paid', 'shipped', 'completed', 'cancelled') DEFAULT 'pending',
    order_date DATE NOT NULL,                       -- 只需要日期
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**文章表**：


```sql
CREATE TABLE articles (
    id INT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
    title VARCHAR(200) NOT NULL,
    content TEXT,                                    -- 文章内容
    summary VARCHAR(500),                            -- 摘要
    view_count INT UNSIGNED DEFAULT 0,               -- 浏览次数
    tags SET('技术', '设计', '产品', '运营'),        -- 标签
    published_at DATETIME,                           -- 发布时间
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 8.3.避免常见错误

1. **不要使用过大的类型**：
   - 错误：使用 `BIGINT` 存储年龄
   - 正确：使用 `TINYINT UNSIGNED`
2. **不要使用浮点数存储金额**：
   - 错误：使用 `FLOAT` 或 `DOUBLE` 存储价格
   - 正确：使用 `DECIMAL(10, 2)` 存储价格
3. **不要使用字符串存储日期**：
   - 错误：使用 `VARCHAR` 存储日期
   - 正确：使用 `DATE` 或 `DATETIME`
4. **不要使用过长的 VARCHAR**：
   - 错误：所有字符串字段都使用 `VARCHAR(65535)`
   - 正确：根据实际需求选择合适的长度
5. **注意字符集影响**：
   - UTF-8 编码下，一个中文字符占用 3 字节
   - 计算存储空间时要考虑字符集

### 8.4.性能优化建议

1. **主键选择**：
   - 优先使用整数类型（`INT` 或 `BIGINT`）
   - 使用 `AUTO_INCREMENT` 自动生成
   - 使用 `UNSIGNED` 扩大正数范围
2. **索引字段**：
   - 尽量使用固定长度的类型（如 `INT`、`CHAR`）
   - 避免在 `TEXT` 和 `BLOB` 类型上创建索引（可以使用前缀索引）
3. **查询优化**：
   - 对于经常用于 `WHERE` 条件的字段，选择合适的数据类型
   - 枚举类型在查询时性能较好
4. **存储优化**：
   - 使用 `NOT NULL` 约束可以节省存储空间
   - 合理使用 `DEFAULT` 值

## 9.总结

本文全面介绍了 MySQL 支持的主要数据类型，包括：

1. **数值类型**：
   - 整数类型：`TINYINT`、`SMALLINT`、`MEDIUMINT`、`INT`、`BIGINT`
   - 浮点数类型：`FLOAT`、`DOUBLE`
   - 定点数类型：`DECIMAL`
   - 位类型：`BIT`
2. **日期时间类型**：
   - `DATE`：存储日期
   - `TIME`：存储时间或持续时间
   - `YEAR`：存储年份
   - `DATETIME`：存储日期和时间
   - `TIMESTAMP`：存储时间戳（带自动更新功能）
3. **字符串类型**：
   - 定长字符串：`CHAR`、`BINARY`
   - 变长字符串：`VARCHAR`、`VARBINARY`
   - 文本类型：`TINYTEXT`、`TEXT`、`MEDIUMTEXT`、`LONGTEXT`
   - 二进制类型：`TINYBLOB`、`BLOB`、`MEDIUMBLOB`、`LONGBLOB`
4. **枚举集合类型**：
   - `ENUM`：存储单一预定义值
   - `SET`：存储多个预定义值
5. **空间数据类型**：
   - `POINT`、`LINESTRING`、`POLYGON` 等，用于存储地理信息

**关键要点**：

- 选择数据类型时要考虑存储空间、性能、精度和扩展性
- 金额等需要精确计算的场景使用 `DECIMAL`，不要使用浮点数
- 日期时间使用专门的日期时间类型，不要使用字符串
- 根据实际需求选择合适的长度，不要过度预留
- 主键优先使用整数类型，配合 `AUTO_INCREMENT` 使用

掌握这些数据类型的特点和适用场景，能够帮助你设计出更高效、更合理的数据库表结构。在实际开发中，要根据具体业务需求灵活选择，并在性能和存储空间之间找到平衡点。
